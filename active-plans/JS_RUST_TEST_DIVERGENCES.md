# JS → Rust: Exhaustive Test-Case Divergences

Every JS test case whose Rust output differs, enumerated. Companion to
[JS_RUST_DIFF.md](JS_RUST_DIFF.md) (the API-level diff); this file is the
empirical, case-by-case output comparison.

## Methodology

- **Rust test suite**: `cargo test --release` — **all pass**. Every fixture that
  Rust *asserts* against JS therefore matches; divergences only exist where Rust
  does **not** assert exact JS output (the ast→string fixtures) or where they are
  explicitly snapshotted (the `*-known-failures` / `*-known-divergences` files).
- **ast→latex / ast→text**: measured by feeding each JS spec fixture's `ast`
  through `from_js → to_latex/to_text` and comparing to the JS `out` string.
  This is exactly what the committed probe
  `packages/math-expressions-rs/tests/zzz_divergence_probe.rs` does — the numbers
  below are reproducible with
  `cargo test --test zzz_divergence_probe -- --nocapture`.
- **Rounding**: JS outputs generated by running `lib/math-expressions.js` in node
  on the `quick_rounding.spec.js` single-number cases; compared to Rust
  `round_numbers_to_precision/decimals`.
- **Math operations** (derivative/evaluate/expand/simplify/ops/equality/
  assumptions/numeric/ode): the Rust corpus tests run thousands of random inputs
  with JS/mathjs as the oracle and **snapshot every divergence**; those snapshot
  files are reproduced verbatim in §5.

## 1. Parsers (text→ast, latex→ast): 0 divergences

Every `text-to-ast.json` (551), `latex-to-ast.json` (617), and the JS-oracle
`*-edge` fixtures are **asserted** by `tests/text_parse.rs` / `tests/latex_parse.rs`
and pass — Rust produces the identical AST tree for every JS parser test case.
Parser trees are the one place the JS output is still the spec.

## 6. Operation specs not string-diffable here

These JS specs assert results via `.equals()` (mathematical equality) or
structural `.tree` equality on operations that Rust intentionally re-canonicalizes
(different display form, same math). They are **not** a clean source of "output
differs" the way the converters are; their correctness is covered by the Rust
suite and the §5 corpora at far larger scale. Listed for completeness:

| JS spec | cases | Rust coverage | Notes |
|---|---:|---|---|
| `quick_arithmetic.spec.js` | 6 | `ops_corpus` (200), `ops.rs`, `number_ops.rs` | `.simplify()` canonical form may differ in tree; math equal |
| `quick_solve.spec.js` | ~9 | `grade.rs` | asserted via `trees.equal` after `evaluate_numbers`; solve_linear present |
| `quick_sets.spec.js` | 7 | `sets.rs` (8) | `.equals()` booleans + `match_partial` scores |
| `slow_matrix.spec.js` | 12 | `matrix.rs` (31) | vector add/sub/scalar-mul tree equality (superset in Rust) |
| `quick_normalization.spec.js` | 62 | `norm.rs` (12) | parse trees (0 divergence, §1) + per-pass normalizers folded into `canonicalize` |
| `quick_transformation.spec.js` | 8 | `expand_corpus` (247) | `expand` covered; **`expand_relations` op absent in Rust** |
| `slow_polynomial.spec.js` | ~23 | none | **entire Groebner/poly engine absent in Rust** — cannot run |
| `quick_pm.spec.js` | 58 | `tests/pm.rs` (14), `pm::tests` (3) | ± primitives, `.equals`, `simplify` rules, and `expand` guards ported (`src/pm.rs`, `eq::pm_equals`, `norm`) |

`slow_polynomial` is not "output differs" — the operation does not exist in Rust
at all (see [JS_RUST_DIFF.md](JS_RUST_DIFF.md) §4). `quick_pm` is fully covered:
the detection/counting/sign-expansion helpers, `.equals` set comparison, the
`simplify` canonicalization rules (reorder, scaling `2·±x → ±(2x)`, `−(±x) → ±x`),
and the `expand` guards that keep a single ± from being duplicated all match JS.

## 7. Which divergences are bugs vs. intentional

**All correctness, ambiguity, and clarity bugs are fixed**
(`tests/formatter_fixes.rs`): invalid double-superscript power towers
(`x^{y}^{z}` → `\left(x^{y}\right)^{z}`), `\lnot`/logical-operator
parenthesization, unescaped LaTeX `%` (a comment-injection bug), the
`\partial  x` double space, `perp` → `⟂` in text, **radicals raised to a power**
(`\left(\sqrt{2}\right)^{3}`), and **units in a product**
(`\left(x \%\right) y`, `\$ x`).

**Remaining divergences are intentional redesign or accepted style** (Rust
formatter is clean-slate; JS strings are no longer the spec):
- Delimiter padding (`( x, y )` → `(x, y)`), `not` → `¬`, negative-fraction
  placement, scientific-notation expansion, sub/superscript associativity parens,
  `angle`/Leibniz spacing conventions.
- One edge case: a malformed 1-argument `nthroot` — JS interprets it as `\sqrt`,
  Rust renders it as an ordinary function application.

The parser (`text/latex → ast`), rounding, and the ops/equality/numeric/ode
corpora show **zero** divergence — those are safe.
