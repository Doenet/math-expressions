# math-expressions-rs-wasm

The WebAssembly boundary for the [`math-expressions`](../math-expressions-rs)
Rust core, for JavaScript consumers — principally **Doenet**, which plots via
**jsxgraph**. This package co-locates two things (the Doenet
`doenetml-worker-rust` layout):

1. **`src-rust/`** (+ `Cargo.toml` at the package root) — the
   `math-expressions-wasm` Rust crate: a thin `wasm-bindgen` adapter over the
   core crate's public API, compiled to a `cdylib`. Build it with `build-wasm.sh`
   (or `npm run build:wasm`), which emits the wasm-bindgen package into `pkg/`
   (git-ignored). Verify the build with `npm test` (the `tests/` end-to-end
   suite).
2. **`src-js/`** — the TypeScript bindings (the AST → math.js bridge below).

`Cargo.toml` sits as a peer of `package.json` at the package root; the two
language trees are separated into `src-rust/` and `src-js/`.

## What's here

The load-bearing TS piece is the **AST → math.js bridge**. The Rust port owns the
AST and its normalization; this package owns only the construction of math.js
nodes (which are JavaScript objects that must be built in JS). A consumer
compiles an expression **once** and then calls `.evaluate(scope)` per sample in
a tight loop that stays entirely in JS — no JS↔WASM boundary crossing per point,
which is what makes numeric graphing fast.

This is "option 1" of the porting analysis: keep AST → math.js in JS/TS, fed the
already-normalized Rust AST, rather than reimplementing math.js in Rust.

## Layout

- `Cargo.toml` — the `math-expressions-wasm` crate manifest (peer of
  `package.json`); a workspace member built for `wasm32-unknown-unknown`, with
  `[lib] path = "src-rust/lib.rs"`.
- `src-rust/*.rs` — the Rust wasm-bindgen sources; see `src-rust/lib.rs` for the
  exported surface. `build-wasm.sh` (package root) compiles it; `tests/` is the
  end-to-end suite that exercises the built `pkg/` via Vitest.
- `pkg/` — wasm-bindgen output (generated by `build-wasm.sh`, git-ignored).
- `tests/end-to-end.test.ts` — loads the **web-target** `pkg/` the browser way
  (ESM `import` + `initSync`, no `require`) and checks a slice of every
  subsystem. Run `npm run build:wasm && npm test`.
- `src-js/tree-to-mathjs.ts` — `Tree` (the JSON AST from `expr.tree_json()`) →
  math.js `MathNode`, plus `factorial → gamma`, `compileTree`, and the
  `rustExprToMathNode` / `compileRustExpr` bridge (normalization done Rust-side
  via `normalize_function_names`).
- `src-js/wasm.ts` — the single source of truth for the wasm handle/module
  types (imported by `math-expressions-js-compat`). The exhaustive types are the
  generated `pkg/math_expressions_wasm.d.ts`; this is the structural surface.
- `src-js/index.ts` — public entry point.

## Usage

```ts
import { create, all } from "mathjs";
import { compileRustExpr } from "math-expressions-rs-wasm";
// `expr` is a parsed handle from the wasm module (parse_text / parse_latex)

const math = create(all, {});
const compiled = compileRustExpr(math, expr); // normalizes Rust-side, builds node
const y = compiled.evaluate({ x: 3.2 });      // call this per sample (jsxgraph)
```

math.js is a **peer dependency** — pass in your own configured instance so it
isn't double-bundled.
